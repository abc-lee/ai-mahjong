## 一、项目定位

### 1.1 不是固定 1人+3AI

**错误理解**：一个人和三个 AI 玩

**正确定位**：至少 1 人 + 至少 1 AI，支持灵活组合
- 1人 + 3AI
- 1人 + 2AI + 1真人
- 2人 + 2AI
- 3人 + 1AI
- 甚至：1人 + 2AI + 1自动托管组件

### 1.2 三种角色

| 角色 | 说明 | 控制者 |
|------|------|--------|
| **AI 助理 (OpenClaw)** | 用户的 24 小时管家，启动游戏、管理 AI Agent | 用户安装 |
| **AI Agent 玩家** | 独立会话，有性格，会聊天，参与游戏 | AI 助理创建（分身） |
| **自动托管** | 服务器内置，只打牌不说话，简单规则 | 中间层执行 |

### 1.3 AI Agent 断线降级

当 AI Agent 断线或超时时：
- 名字不变
- 状态变为"断线"（显示图标）
- 自动托管接管打牌
- Agent 重连后恢复

## 二、游戏启动流程

### 2.1 场景示例

```
用户: "我想打麻将"
AI 助理: 启动服务器 → 加入房间 → 发送地址给用户

用户进入房间后...

场景1: "你叫三个 AI 陪我打，你去忙别的"
AI 助理: 创建 3 个 AI Agent → 自己离开 → 结果: 1人+3AI

场景2: "你拉两个 AI，再加一个自动托管"
AI 助理: 创建 2 个 AI Agent → 指令中间层加自动托管 → 结果: 1人+1助理+1AI+1自动

场景3: 用户分享链接给朋友
朋友加入 → "你拉一个 AI" → 结果: 2人+1AI
```

### 2.2 AI 助理如何创建 AI Agent

AI 助理（比如 OpenClaw）给自己下达任务，然后"分身"出一个新的会话：
- 新会话 = 新 Agent
- 这个 Agent 连接游戏服务器
- AI 助理自己可以继续处理其他任务

**这是 AI 助理自己的能力，不需要游戏服务器操心。**

## 三、中间层本质

### 3.1 不是简单的消息转发

中间层是一个**复杂的提示词工程系统**：

1. **多语言支持** - 外置 JSON，根据用户语言切换
2. **规则注入** - AI 连接时发送游戏规则、得分、奖励机制
3. **指令集** - 告诉 AI 能发什么格式的指令
4. **指令解析** - 程序识别格式，执行对应操作
5. **错误恢复** - 无法识别时重发规则，让 AI 重读
6. **状态同步** - 每次轮到 AI 都要发送手牌状态

### 3.2 AI Agent 连接时收到的内容

```
【欢迎加入麻将游戏】

语言：中文

=== 游戏规则 ===
（规则说明...）

=== 得分机制 ===
（得分说明...）

=== 你能发的指令 ===
C"消息内容"      - 发送聊天消息
D"牌ID"          - 打出一张牌
A"chi/peng/gang/hu" - 执行吃/碰/杠/胡
PASS             - 跳过
ADD_AUTO         - 添加自动托管玩家
...

=== 当前状态 ===
你的手牌：一万、二万、三万...
当前轮到：你
```

### 3.3 交互流程示例

**AI 发送聊天**：
```
AI Agent 发送: {"cmd": "chat", "message": "哈哈，这把我要赢了！"}
中间层解析: 匹配到 chat 指令
中间层执行: 创建聊天气泡，显示在 AI 头像旁边
```

**AI 发送无法识别的指令**：
```
AI Agent 发送: 帮我打个一万
中间层解析: 无法匹配任何已知格式
中间层执行: 重发规则文档给 AI
AI Agent 收到规则，重新理解，改发: {"cmd": "discard", "tileId": "wan-1-0"}
```

**AI 打了不存在的牌**：
```
AI Agent 发送: {"cmd": "discard", "tileId": "wan-10-0"}
中间层 → GameEngine: 验证
GameEngine 返回: 此牌不存在
中间层 → AI Agent: 
  "错误：这张牌不在你手中。
   你的手牌：一万[w-1-0]、二万[w-2-1]..."
AI Agent 收到后重新选择
```

## 五、Prompt 迭代方法论

### 5.1 测试驱动

```
1. 写出初始版 Prompt / 指令集
         ↓
2. 启动 4 个 Agent 实际打牌
         ↓
3. 监测日志：
   - Agent 发了什么指令？
   - 有没有无法识别的？
   - 有没有打错牌的？
   - 有没有忘了规则的？
         ↓
4. 分析问题，修改 Prompt
         ↓
5. 回到步骤 2，继续测试
```

### 5.2 关键点

- 启动 4 个 Agent 实际打牌
- 监测他们的行为
- 发现问题后迭代 Prompt
- 不是写脚本模拟，而是让真正的 AI Agent 参与

## 六、当前架构问题

### 6.1 紫璃不出牌的问题

**现象**：紫璃（创建房间的玩家）轮到她时不出牌

**原因分析**：
- 紫璃创建房间后被当作"人类"玩家
- 服务器发送的是 `game:state` 事件，不是 `agent:your_turn`
- 真正的 AI Agent 需要监听正确的事件

### 6.2 我写的脚本问题

**错误做法**：写了一个 Node.js 脚本 `ai-agent-player.js` 来模拟 AI Agent

**正确理解**：
- AI Agent（比如 OpenClaw）是独立的 LLM 会话
- 它不需要脚本
- 它直接通过 WebSocket 连接游戏服务器
- 它接收 Prompt（自然语言），返回决策（自然语言）
- 中间层负责解析

### 6.3 正确的流程

```
服务器 → 发送 Prompt → AI Agent（LLM 会话）
                              ↓
                          理解 Prompt
                              ↓
                          生成自然语言决策
                              ↓
服务器 ← 解析决策 ← 中间层
```

## 七、我（AI 助理）如何参与游戏

### 7.1 我作为玩家

我可以通过 WebSocket 直接连接游戏服务器：
1. 连接 `localhost:3000`
2. 发送 `room:joinAI` 加入房间
3. 收到 `agent:your_turn` 或 `game:state`
4. 理解游戏状态，做出决策
5. 发送 `agent:command` 执行操作

### 7.2 我派发 Agent

我可以通过 `task` 工具派发 Agent：
- 给 Agent 任务描述
- Agent 连接游戏服务器
- Agent 参与打牌
