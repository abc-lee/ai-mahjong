# AI 玩家架构设计

> 版本：v4.0
> 更新时间：2026-03-10
> 核心理念：AI Agent 作为真正的玩家，通过 WebSocket 连接游戏，享受娱乐陪伴体验
> 使用场景：私人游戏，主Agent作为用户的个人AI助理

---

## 1. 这不是传统游戏

**传统游戏**：服务器内置 AI 逻辑 → AI 是 NPC

**这个项目**：AI Agent 作为真正的玩家 → AI 有独立会话、独立性格、独立决策

用户说"准备打游戏"时，主Agent（如 OpenCode/OpenClaw）：
1. 启动游戏服务器
2. 返回链接给用户
3. 用户点击链接进入游戏，选择座位
4. 用户可以通过独立通话渠道让主Agent派发AI（紫璃、白泽、李瞳...）
5. 每个 AI Agent 有独立的会话、独立的性格
6. 他们会吵架、吐槽、情绪化

**用户获得的体验**：和 AI 一起玩，看他们互怼，享受娱乐陪伴的过程。

## 2. 完整架构

```
┌─────────────────────────────────────────────────────────────┐

│                        WebSocket                             │

                              ↑↓
```

## 3. 为什么这样设计

### 3.1 人类玩家需要什么

图形界面 —— 看得见牌、点得着按钮。

### 3.2 AI Agent 需要什么

**不是图形界面，而是 Prompt。**

图形界面对 AI 来说效率很低——要解析 UI、理解状态、做决策。

AI 需要的是：
- 中间层直接把游戏状态翻译成 Prompt 发过来
- AI 理解后，直接发决策 JSON 回去
- 中间层验证规则，告诉 AI 成功还是失败

### 3.3 为什么游戏服务器必须知道玩家类型

因为人类和 AI 需要的数据格式不同：
- 人类 → 图形界面数据
- AI → Prompt 文本

### 3.4 AIAdapter 的降级作用

当 AI Agent 断线、超时、反应太慢时，AIAdapter 自动接管：
1. 规则引擎决策
2. 随机决策

保证游戏永远不会因为 AI 而卡住。

## 4. 玩家类型定义

```typescript
interface Player {
  id: string;
  name: string;
  position: 0 | 1 | 2 | 3;   // 座位：东南西北
  
  // 核心字段：区分三种玩家类型
  type: 'human' | 'ai-agent' | 'npc';
  
  // 人类玩家
  socketId?: string;
  isOnline: boolean;
  
  // AI Agent 玩家
  agentId?: string;           // Agent ID
  aiConfig?: AIConfig;
  
  // 通用游戏数据
  hand: Tile[];
  melds: Meld[];
  discards: Tile[];
  isDealer: boolean;
  score: number;
  isReady: boolean;
}
```

### 三种玩家类型

| 类型 | 标识 | 来源 | 特点 |
|------|------|------|------|
| 人类玩家 | `human` | 浏览器连接 | 看图形界面，点按钮操作 |
| AI Agent 玩家 | `ai-agent` | 主Agent派发的子Agent | 收 Prompt，发 JSON 指令，有性格会聊天 |
| NPC | `npc` | 服务器内置 | 简单规则决策，只打牌不说话 |

```typescript
interface AIConfig {
```

## 7. Prompt 设计

详见 `ai-agent-prompt-design.md`。

核心原则：
1. AI 收到的是 **Prompt 文本**，不是图形界面数据
2. AI 返回的是 **JSON 决策**，不是点击操作
3. Prompt 必须包含 AI 做决策所需的全部信息
